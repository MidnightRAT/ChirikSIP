# Глибокий рев'ю ChirikSIP

## Загальна інформація

- **Проєкт**: ChirikSIP — мінімалістичний SIP-клієнт для KDE Plasma та Windows
- **Мова**: C++17, Qt6, PJSIP 2.13.1, PortAudio 19.7
- **Розмір**: ~1958 рядків C++, 15 файлів (8 .cpp + 7 .h)

## Знайдені проблеми

### CRITICAL

#### 1. Потокова безпека PJSIP callback'ів
**Файл**: `src/sipclient.cpp`

PJSIP callback'и (`onRegState2`, `onCallState`, `onCallMediaState`, `onIncomingCall`) викликаються з **внутрішнього потоку PJSIP**, але безпосередньо читають/пишуть члени Qt-об'єктів та emit'ять сигнали без синхронізації:

- **Рядок 345**: `self->m_incomingCallId = callId;` — запис з callback-потоку, читання з UI-потоку в `answerCall()` (рядок 202). Ні mutex, ні atomic, ні signal marshalling.
- **Рядки 284-292**: `onCallState` отримує доступ до `self->m_ringtone` та `self->m_audioBridge` з callback-потоку, тоді як головний потік може одночасно викликати `shutdown()` (рядок 99-114) або `answerCall()` (рядок 217-219). **Data race**.
- **Рядки 309-324**: `onCallMediaState` створює `AudioBridge`, призначає `self->m_audioBridge` та викликає `pjsua_conf_connect` — все з неголовного потоку.
- **Рядок 279**: `self->m_audioBridge = nullptr;` в `onCallState` конфліктує з присвоєнням в `onCallMediaState`.

**Вплив**: Краші, use-after-free, зіпсований стан при навантаженні або несприятливому таймінгу.

**Рекомендація**: Всі PJSIP callback'и повинні використовувати `QMetaObject::invokeMethod(self, ..., Qt::QueuedConnection)` для маршрутизації роботи в UI-потік, або використовувати mutex навколо спільного стану.

---

#### 2. Подвійне завершення PortAudio
**Файли**: `src/ringtone.cpp`, `src/audiobridge.cpp`

Обидва модулі незалежно викликають `Pa_Initialize()` / `Pa_Terminate()`. Референс-рахунок PortAudio ненадійний між реалізаціями. Якщо рингтон грає коли дзвінок приймається (AudioBridge відкривається):

1. `Ringtone::start()` → `Pa_Initialize()` (ref=1)
2. `AudioBridge::open()` → `Pa_Initialize()` (ref=2)
3. `Ringtone::stop()` → `Pa_Terminate()` (ref=1 або 0)
4. `AudioBridge::close()` → `Pa_Terminate()` (ref=0 або -1)

Другий `Pa_Terminate()` може знищити stream іншого модуля.

**Вплив**: Краш аудіо-підсистеми при прийнятті дзвінка.

**Рекомендація**: Централізувати управління життєвим циклом PortAudio (singleton або init-рахунок з блокуванням).

---

### HIGH

#### 3. Переповнення буфера в audio callback
**Файл**: `src/audiobridge.cpp`

- **Рядок 38 (header)**: `float m_captureBuffer[160]; float m_playbackBuffer[160];` — фіксований розмір 160 float.
- **Рядок 147**: `std::memcpy(self->m_captureBuffer, in, frameCount * sizeof(float));` — **без перевірки меж**. Якщо PortAudio доставить більше 160 фреймів — переповнення буфера (corrupt сусідні члени: `m_playbackReady` тощо).

`FRAMES_PER_BUFFER = 160` — це лише *рекомендований* розмір; PortAudio може доставити більше або менше.

**Вплив**: Memory corruption в realtime audio callback.

**Рекомендація**: `if (frameCount > FRAMES_PER_BUFFER) frameCount = FRAMES_PER_BUFFER;` на початку `paCallback`.

---

#### 4. Потокова безпека в audio callback
**Файл**: `src/audiobridge.cpp`

`paCallback` (рядки 146-162) працює в потоці PortAudio. Читає/пише `m_captureReady`, `m_captureBuffer`, `m_playbackBuffer`, `m_playbackReady` — неатомічні змінні, спільні з `putFrame`/`getFrame` (потік PJSIP media). Ні memory barriers, ні атоміків, ні блокувань.

Це UB в C++ (data race на неатомічних змінних між потоками). На x86 може "працювати", на ARM або з агресивною оптимізацією — torn reads/writes.

**Вплив**: Артефакти звуку або undefined behavior.

#### 6. Відсутність валідації SIP URI / санітизації введення
**Файл**: `src/sipclient.cpp`

- `makeCall()` (рядки 165-198): мінімальна перевірка `@`, зрізання `sip:` префіксу. Ні перевірки формату SIP URI (RFC 3261), ні екранування спецсимволів, ні max length.
- `registerAccount()` (рядки 122-163): `server`, `username`, `password` використовуються безпосередньо в побудові SIP URI.

#### 7. Витік акаунтів при повторній реєстрації
**Файли**: `src/sipclient.cpp:122-163`, `src/mainwindow.cpp:574`

Кожен виклик `registerAccount()` створює **новий** pjsua account через `pjsua_acc_add`. Попередні акаунти ніколи не видаляються. При зміні налаштувань та повторній реєстрації акаунти накопичуються.

**Рекомендація**: Викликати `pjsua_acc_del(m_accId)` перед `pjsua_acc_add`, або використовувати `pjsua_acc_modify`.

## Архітектурні спостереження

1. **Однопоточність з callback'ами**: PJSIP callback'и викликаються в потоці PJSIP, але emit'ять Qt-сигнали напряму — це основне джерело критичних багів.
2. **SipClient = протокол + диспетчер подій**: Тісне з'єднання PJSIP внутрішніх з Qt signal emission. Чистіший дизайн — відокремити PJSIP event loop від Qt event loop через чергу подій.
3. **AudioBridge як custom pjmedia_port**: Дає повний контроль над конверсією int16<->float, але ускладнює управління життєвим циклом.
4. **Статичні callback'и з user_data**: Стандартний підхід PJSIP, але крихкий — null-check захищає від крашу, але не від тихого збою (аудіо не відкриється без зворотного зв'язку користувачу).

## Верифікація

1. `cmake -B build && cmake --build build` — збірка без помилок та попереджень
2. Запуск на Linux — реєстрація на SIP-сервері, вхідний/вихідний дзвінок
3. Перевірка thread safety: запуск Valgrind/Helgrind або ThreadSanitizer (`-fsanitize=thread`)
4. Перевірка Memory Sanitizer: `-fsanitize=address,undefined`
5. Тест PortAudio: прийняття дзвінка під час рингтону — не повинно бути крашу
